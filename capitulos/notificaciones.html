Jetpack Compose · Android

Cómo crear Notificaciones en Android

Guía completa sobre permisos, canales, AlarmManager y BroadcastReceiver con Jetpack Compose.

01

Permisos necesarios

Para trabajar con notificaciones en Android es necesario declarar los siguientes permisos en el archivo AndroidManifest.xml:

uses-permission
POST_NOTIFICATIONS
Requerido desde Android 13 (API 33) para mostrar notificaciones al usuario.
uses-permission
SCHEDULE_EXACT_ALARM
Permite programar alarmas exactas con AlarmManager en Android 12 y superior.
uses-permission
RECEIVE_BOOT_COMPLETED
Permite recibir el evento de arranque del dispositivo para reprogramar alarmas.
AndroidManifest.xml 
<uses-permission android:name="android.permission.POST_NOTIFICATIONS"/>
<uses-permission android:name="android.permission.SCHEDULE_EXACT_ALARM"/>
<uses-permission android:name="android.permission.RECEIVE_BOOT_COMPLETED"/>

Solicitar permiso en tiempo de ejecución

A partir de Android 13 (API 33 — TIRAMISU), el permiso POST_NOTIFICATIONS debe solicitarse al usuario en tiempo de ejecución. El flujo es el siguiente:

Verificar versión Android ≥ 13
→
checkSelfPermission()
→
¿Concedido?
→
requestPermission.launch()
MainActivity.kt 
// Registra el lanzador de solicitud de permisos
val requestPermission = registerForActivityResult(
    ActivityResultContracts.RequestPermission()
) { /* manejar resultado */ }

fun pedirPermiso() {
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
        if (ActivityCompat.checkSelfPermission(
                this,
                Manifest.permission.POST_NOTIFICATIONS
            ) != PackageManager.PERMISSION_GRANTED
        ) {
            requestPermission.launch(Manifest.permission.POST_NOTIFICATIONS)
        }
    }
}
ℹ️
¿Qué devuelve checkSelfPermission()? El método devuelve un entero: 0 si el permiso fue concedido, -1 si fue denegado. La constante PackageManager.PERMISSION_GRANTED equivale a 0, lo que facilita la comparación.
02

Canal de notificaciones

A partir de Android 8.0 (API 26 — Oreo), es obligatorio crear un canal de notificaciones antes de mostrar cualquier notificación. Un canal es una categoría que permite a Android agrupar notificaciones similares y al usuario configurar individualmente el sonido, la vibración y la prioridad.

✅
El canal se crea una sola vez Si el canal ya existe, Android lo ignora. Es seguro llamar a createNotificationChannel() cada vez que se inicia la app.
NotificationHelper.kt 
fun createChannel(context: Context) {
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O) {

        val channel = NotificationChannel(
            CHANNEL_ID,
            "Notificaciones",
            NotificationManager.IMPORTANCE_HIGH
        ).apply {
            description = "Canal de alertas y recordatorios"
        }

        // Obtenemos el servicio del sistema para gestionar notificaciones
        val manager = context.getSystemService(NotificationManager::class.java)
        manager.createNotificationChannel(channel)
    }
}

Crear y mostrar una notificación

Para construir una notificación se utiliza NotificationCompat.Builder, que recibe el contexto y el ID del canal. Una vez construida, se muestra mediante NotificationManagerCompat.notify().

NotificationHelper.kt 
@SuppressLint("MissingPermission")
fun createNotification(context: Context, myNotification: MyNotification) {

    val notification = NotificationCompat.Builder(context, CHANNEL_ID)
        .setSmallIcon(android.R.drawable.star_on)
        .setContentTitle(myNotification.title)
        .setContentText(myNotification.description)
        .setAutoCancel(true)     // se elimina al tocarla
        .setPriority(NotificationCompat.PRIORITY_HIGH)
        .build()

    NotificationManagerCompat
        .from(context)
        .notify(myNotification.requestCode, notification)
}
03

AlarmManager

AlarmManager es el servicio del sistema que permite programar una acción para ejecutarse en un momento específico en el futuro. Cuando llega el tiempo programado, dispara un PendingIntent que normalmente apunta a un BroadcastReceiver.

→
Tiempo cumplido
→
PendingIntent
→
BroadcastReceiver
→
Notificación

Verificar permiso de alarmas exactas

En Android 12 (API 31) y superior, la app debe verificar si tiene permiso para programar alarmas exactas. Si no lo tiene, se redirige al usuario a la pantalla de configuración del sistema.

AlarmManagerRepository.kt 
fun programarNotificacion(context: Context, myNotification: MyNotification) {

    val alarmManager = context.getSystemService(Context.ALARM_SERVICE) as AlarmManager

    // Verificar permiso en Android 12+
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.S) {
        if (!alarmManager.canScheduleExactAlarms()) {
            val intent = Intent(Settings.ACTION_REQUEST_SCHEDULE_EXACT_ALARM)
            intent.flags = Intent.FLAG_ACTIVITY_NEW_TASK
            context.startActivity(intent)
            return
        }
    }

    // Crear el Intent hacia el BroadcastReceiver
    val intent = Intent(context, NotificationBroadcast::class.java).apply {
        putExtra("notification", Json.encodeToString(myNotification))
    }

    val pendingIntent = PendingIntent.getBroadcast(
        context,
        myNotification.requestCode,
        intent,
        PendingIntent.FLAG_IMMUTABLE or PendingIntent.FLAG_UPDATE_CURRENT
    )

    // Programar la alarma exacta
    alarmManager.setExactAndAllowWhileIdle(
        AlarmManager.RTC_WAKEUP,
        myNotification.time,
        pendingIntent
    )
}

Flags del PendingIntent

FLAG_IMMUTABLE
El PendingIntent no puede modificarse después de crearse. Obligatorio en Android 12+.
FLAG_UPDATE_CURRENT
Si ya existe un PendingIntent con el mismo ID, actualiza sus datos en lugar de crear uno nuevo.
💡
¿Qué es RTC_WAKEUP? AlarmManager.RTC_WAKEUP dispara la alarma basándose en la hora real del reloj del sistema y despierta el dispositivo aunque esté en modo Doze o con la pantalla apagada. Es el tipo correcto para notificaciones que deben llegar en un momento preciso.
04

BroadcastReceiver

El BroadcastReceiver es el componente que recibe el PendingIntent cuando el AlarmManager lo dispara. Aquí se recuperan los datos de la notificación y se muestra al usuario.

NotificationBroadcast.kt 
class NotificationBroadcast : BroadcastReceiver() {

    override fun onReceive(context: Context, intent: Intent) {
        val notificationString = intent.getStringExtra("notification")

        notificationString?.let {
            val notification = Json.decodeFromString<MyNotification>(it)
            NotificationHelper.createNotification(context, notification)
        }
    }
}
⚠️
El BroadcastReceiver debe declararse en el Manifest Sin esta declaración, Android no reconocerá el receptor y la notificación nunca se mostrará.
AndroidManifest.xml 
<receiver
    android:name=".NotificationBroadcast"
    android:exported="true">
    <intent-filter>
        <action android:name="android.intent.action.BOOT_COMPLETED"/>
    </intent-filter>
</receiver>
05

Serialización del objeto

Para enviar el objeto MyNotification a través del PendingIntent, es necesario convertirlo a una cadena de texto en formato JSON. Para esto se utiliza la librería kotlinx.serialization.

Json.encodeToString() convierte un objeto Kotlin a una cadena en formato JSON. Json.decodeFromString() hace el proceso inverso: convierte la cadena JSON de vuelta al objeto original.

MyNotification.kt 
@Serializable
data class MyNotification(
    val title: String = "",
    val description: String = "",
    val time: Long = 0L,
    val requestCode: Int = 0,
    val route: String = ""
)

// Serializar: objeto → JSON
val json = Json.encodeToString(myNotification)

// Deserializar: JSON → objeto
val notification = Json.decodeFromString<MyNotification>(json)
